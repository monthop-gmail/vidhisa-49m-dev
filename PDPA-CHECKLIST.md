# PDPA Checklist — ระบบวิทิสา 49 ล้านนาที

- **อ้างอิงกฎหมาย:** พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 (PDPA)
- **วันที่จัดทำ:** 2026-07-11
- **สถานะ:** ร่างสำหรับทีมพิจารณา (ยังไม่ลงมือแก้)
- **เกี่ยวข้องกับ:** `SECURITY-AUDIT.md` (ประเด็นความปลอดภัยเชิงเทคนิค)

> เอกสารนี้เป็นการประเมินเบื้องต้นจากมุมผู้พัฒนา จากการอ่านซอร์สโค้ด **ไม่ใช่คำแนะนำทางกฎหมาย** — ควรให้ที่ปรึกษากฎหมาย/DPO ตรวจสอบก่อนสรุปแนวทางปฏิบัติจริง

---

## ข้อมูลส่วนบุคคล (PII) ที่ระบบเก็บอยู่ปัจจุบัน

| ตาราง | ฟิลด์ PII | หมายเหตุ |
|-------|-----------|----------|
| `participants` | ชื่อ-สกุล, เพศ, อายุ, ตำบล/อำเภอ/จังหวัด, เบอร์โทร, LINE ID | ข้อมูลระบุตัวตนหลัก |
| `organizations` | ชื่อผู้ติดต่อ, เบอร์โทร, LINE ID | ผู้ประสานงานองค์กร |
| `branches` | เบอร์โทร admin1/2/3 | ผู้ดูแลสาขา |
| `records` | `ip_address`, `device_id`, `submitted_phone` | เก็บเพื่อ anti-fraud |
| `branch_view_logs` | `ip`, `user_agent` | log การเข้าถึง me-ui |
| `users` | ชื่อ, email, เบอร์โทร | บัญชีผู้ดูแลระบบ |

**Consent ปัจจุบัน:** มีเพียง `participants.privacy_accepted` (boolean) — ไม่มีเวอร์ชัน, วันเวลา, หรือแหล่งที่มาของการยินยอม

---

## 1. มาตรการความมั่นคงปลอดภัย (มาตรา 37) — 🔴 เร่งด่วน

> เชื่อมโยงกับ `SECURITY-AUDIT.md` ข้อ C-1, C-2

- [ ] **ปิดช่อง PII รั่ว** — endpoint อ่านข้อมูล participant/record เข้าถึงได้โดยไม่ต้อง auth (C-1) ถือเป็นการละเมิดมาตรา 37 โดยตรง ต้องเพิ่ม authentication
- [ ] **ปิดช่องแก้ไขข้อมูลโดยไม่มีสิทธิ** (C-2)
- [ ] จำกัดสิทธิเข้าถึง PII ตามบทบาท (branch admin เห็นเฉพาะสาขาตน)

**ผลถ้าไม่ทำ:** เป็นการละเมิด PDPA ที่ลงโทษได้ (ปรับทางปกครองสูงสุด 5 ล้านบาท) นอกเหนือจากความเสี่ยงด้านความปลอดภัย

---

## 2. การขอความยินยอม (Consent — มาตรา 19) — 🟠 สูง

ปัจจุบันเก็บแค่ boolean ซึ่งพิสูจน์ไม่ได้ว่าขอยินยอมอย่างไร/เมื่อไหร่

- [ ] เพิ่มฟิลด์ `consent_version` — เวอร์ชันนโยบายที่ผู้ใช้กดยอมรับ (แก้ policy = ขอใหม่)
- [ ] เพิ่ม `consent_at` (timestamp) + `consent_source` (web / ggs-import / สาขากรอกให้)
- [ ] ข้อความ consent ต้องแยกตามวัตถุประสงค์ ระบุชัด ไม่ใช่ checkbox เดียวเหมารวม
- [ ] **กรณี import จาก Google Sheet** (`ggs.py:583,848` ตั้ง `privacy_accepted=True` อัตโนมัติ) — ต้องมีหลักฐานว่าผู้เข้าร่วมยินยอมจริงในต้นทาง ไม่ใช่ระบบตั้งให้เอง
- [ ] consent ต้อง "แยกออกจากเงื่อนไขบริการอื่น" และถอนได้ง่ายเท่าที่ให้

---

## 3. สิทธิของเจ้าของข้อมูล (มาตรา 30–36) — 🟠 สูง

ระบบต้องมีกลไกให้ใช้สิทธิได้จริง:

- [ ] **สิทธิเข้าถึง / ขอสำเนา** (ม.30) — me-ui ดูยอดตัวเองได้บางส่วนแล้ว ควรให้ดู/export ข้อมูลตัวเองครบถ้วน
- [ ] **สิทธิขอลบ / ทำลาย** (ม.33) — ปัจจุบัน **ไม่มี** endpoint ลบ participant ต้องมีขั้นตอนลบหรือ anonymize เมื่อมีคำขอ
- [ ] **สิทธิแก้ไขให้ถูกต้อง** (ม.35)
- [ ] **สิทธิคัดค้าน / เพิกถอนความยินยอม** (ม.32, 34) — ต้องมีช่องทางถอน consent แล้วหยุดประมวลผล
- [ ] **สิทธิให้โอนย้ายข้อมูล** (ม.31)
- [ ] กำหนดช่องทางและระยะเวลาตอบสนองคำขอ (ปกติภายใน 30 วัน)

---

## 4. เก็บข้อมูลเท่าที่จำเป็น + ระยะเวลาเก็บ (มาตรา 22, 23) — 🟡 กลาง

- [ ] กำหนด **retention period** ให้ `records.ip_address`, `records.device_id`, `branch_view_logs.ip/user_agent` แล้วลบอัตโนมัติเมื่อพ้นกำหนด (ปัจจุบันเก็บถาวร)
- [ ] ทบทวนว่า `line_id`, เบอร์โทร จำเป็นต่อวัตถุประสงค์โครงการจริงหรือไม่
- [ ] กำหนดว่าเมื่อโครงการจบ (deadline) จะจัดการข้อมูลอย่างไร

---

## 5. เอกสาร / กระบวนการ (กฎหมายบังคับ ไม่ใช่โค้ด) — 🟡 กลาง

- [ ] **นโยบายความเป็นส่วนตัว (Privacy Notice)** — หน้าเว็บแสดง: เก็บอะไร, เพื่ออะไร, เก็บนานแค่ไหน, แชร์ให้ใคร, ช่องทางใช้สิทธิ (ปัจจุบันมีแค่ checkbox ไม่พบหน้านโยบายจริง)
- [ ] **บันทึกกิจกรรมการประมวลผล (RoPA — ม.39)**
- [ ] **ระบุบทบาท** ผู้ควบคุมข้อมูล (โครงการ) / ผู้ประมวลผล (Google Sheets, Cloudflare, ผู้ให้บริการ SMTP) และจัดทำ **ข้อตกลงประมวลผลข้อมูล (DPA)** กับผู้ประมวลผลภายนอก
- [ ] **แผนรับมือข้อมูลรั่วไหล** (ม.37(4)) — แจ้งสำนักงานคณะกรรมการคุ้มครองข้อมูลส่วนบุคคล (สคส.) ภายใน 72 ชม.
- [ ] พิจารณาแต่งตั้ง **DPO** (เจ้าหน้าที่คุ้มครองข้อมูล) หากเข้าเงื่อนไข ม.41

---

## 6. ความปลอดภัยเชิงเทคนิคเสริม — 🟡 กลาง

- [ ] **เข้ารหัสข้อมูลขณะพัก (at rest)** — พิจารณาเข้ารหัส DB volume หรือ column ที่ sensitive
- [ ] **บังคับ HTTPS/TLS** — มี Cloudflare Tunnel รองรับอยู่แล้ว
- [ ] **Audit log การเข้าถึง PII ฝั่ง admin** — ปัจจุบัน log เฉพาะ me-ui (`branch_view_logs`) ควร log ฝั่ง admin ด้วยว่าใคร export/ดู PII ของใคร เมื่อไหร่
